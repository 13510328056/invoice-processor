# 发票批处理工具软件需求规格说明书

## 1\. 引言

### 1\.1 编写目的

本文档详细描述**发票批处理工具**的功能需求与非功能需求，为开发、测试、验收提供依据。本工具自动化完成电子发票的批量识别、信息提取与结构化导出。

### 1\.2 项目背景

财务人员日常需处理大量电子发票，人工录入效率低下、易出错。本工具通过 OCR 与 AI 技术，实现发票的批量自动化处理，大幅提升财务工作效率。

### 1\.3 术语定义

|术语|定义|
|---|---|
|OFD|Open Fixed-layout Document，中国国家标准 GB/T 33993 定义的版式文档格式，本质为 ZIP 压缩包内嵌 XML 结构化数据|
|OCR|光学字符识别（Optical Character Recognition），将图片/扫描件中的文字转换为可编辑文本|
|XPath|XML 路径语言，用于在 XML 文档中定位和提取节点数据的查询语法|
|LLM|大语言模型（Large Language Model），本系统中用于 OCR 置信度不足时的字段兜底提取|


### 1\.4 适用范围

- 适用用户：财务人员、行政人员、报销管理员

- 适用场景：日常报销发票处理、月度发票统计、发票信息归档

## 3\. 功能需求

### 3\.1 文件扫描模块

#### 3\.1\.1 递归文件夹扫描

- **输入**：用户指定的根文件夹路径

- **处理逻辑**：

    1. 遍历根文件夹及其所有子文件夹

    2. 收集所有文件的完整路径（相对于用户指定的根文件夹）

    3. 不区分大小写识别文件后缀

    4. **跳过忽略目录**：根据 `config.yaml` 的 `skip_directories` 配置跳过指定目录（如 `__pycache__`、`node_modules`、`.git` 等）

    5. **业务去重**（可选）：当 `enable_dedup: true` 时，在字段提取完成后，以**「发票代码 + 发票号码」**为业务去重键，同一键值仅保留首次成功提取的记录，后续重复发票跳过并在日志中记录"跳过重复发票（发票代码:xxx 发票号码:xxx）"；未开启去重时所有发票均保留

- **输出**：所有文件的路径列表

#### 3\.1\.2 文件格式分类

- **筛选规则**：

    |    文件后缀|    分类|    处理方式|
    |---|---|---|
    |    `.ofd`|    电子发票文件|    XML 结构化直接解析（首选），解析失败降级至 OCR|
    |    `.pdf`|    电子发票文件|    OCR 识别处理|
    |    `.jpg`、`.jpeg`|    电子发票文件|    OCR 识别处理|
    |    `.png`|    电子发票文件|    OCR 识别处理|
    |    其他所有格式|    非电子发票文件|    跳过识别，仅记录|

> **约束**：文件格式判断默认仅依据后缀名。可选启用 Magic Number 文件头校验（通过 config.yaml 配置），校验文件头标识以对抗后缀名伪造。OFD 文件的 Magic Number 为 ZIP 格式头（`50 4B 03 04`），与标准 ZIP 一致，需结合后缀名和内部 XML 结构双重验证

### 3\.2 发票识别与提取模块

#### 3\.2\.1 OFD 格式直接解析（首选路径）

**原理**：OFD（Open Fixed-layout Document）是中国国家标准 GB/T 33993 定义的版式文档格式，其本质为 ZIP 压缩包内嵌 XML 结构化数据。电子发票的 OFD 文件包含完整的结构化发票信息，可直接解析提取，无需 OCR。

**处理流程**：

1. **解包**：将 `.ofd` 文件作为 ZIP 压缩包解压
2. **入口定位**：读取 `OFD.xml` 获取文档入口，定位 `Document.xml` 
3. **发票数据提取**：通过 XPath 直接提取结构化字段（如发票代码、号码、金额、购销方信息等）
4. **附件处理**：提取 OFD 包内包含的发票预览图（如有），用于可视化展示

**字段映射规则**：

OFD 发票 XML 节点与输出字段存在标准映射关系。以国家税务总局 OFD 电子发票规范为基础，定义 XPath 映射表：

| 输出字段 | OFD XML XPath 示例 | 备注 |
|---|---|---|
| 发票代码 | `//Invoice/InvoiceCode` | 直接取值 |
| 发票号码 | `//Invoice/InvoiceNo` | 直接取值 |
| 开票日期 | `//Invoice/IssueDate` | 格式统一为 YYYY-MM-DD |
| 校验码 | `//Invoice/CheckCode` | 去除中间空格 |
| 发票类型 | `//Invoice/InvoiceType` | 映射为中文描述 |
| 购买方名称 | `//Invoice/BuyerName` | 直接取值 |
| 购买方纳税人识别号 | `//Invoice/BuyerTaxID` | 直接取值 |
| 购买方地址电话 | `//Invoice/BuyerAddressTel` | 直接取值 |
| 购买方开户行及账号 | `//Invoice/BuyerBankAccount` | 直接取值 |
| 销售方名称 | `//Invoice/SellerName` | 直接取值 |
| 销售方纳税人识别号 | `//Invoice/SellerTaxID` | 直接取值 |
| 销售方地址电话 | `//Invoice/SellerAddressTel` | 直接取值 |
| 销售方开户行及账号 | `//Invoice/SellerBankAccount` | 直接取值 |
| 价税合计（大写） | `//Invoice/TotalAmountCn` | 直接取值 |
| 价税合计（小写） | `//Invoice/TotalAmount` | 去除符号及千位分隔符 |
| 不含税金额 | `//Invoice/PreTaxAmount` | 去除符号及千位分隔符 |
| 税额 | `//Invoice/TaxAmount` | 去除符号及千位分隔符 |
| 商品明细 | `//Invoice/GoodsList/Goods` | 逐条提取转为 JSON |

> **映射动态配置**：上述 XPath 映射关系可外部化至 `config.yaml` 的 `ofd.mapping_path`，支持不同开票系统间的 OFD 格式差异适配，无需修改代码

**降级策略**：当 XML 解析失败（如 OFD 结构不符合规范、XML 损坏、加密等情况），自动降级至 OCR 识别流程，与 PDF/JPG/PNG 共用同一 OCR 管线

**准确率预期**：
- 标准 OFD 发票 XML 解析：字段提取准确率 ≈ 100%（数据直接来源于结构化信息）
- OFD→OCR 降级：与清晰扫描件标准一致（≥ 85%）

#### 3\.2\.2 字段提取策略（多级设计）

从 OCR 引擎输出的文字识别结果中，采用**多级提取策略**：

1. **第一级：正则 + 关键字匹配** — 优先使用，覆盖常见版式
2. **第二级：上下文关联定位** — 当正则失效时，根据字段间层次关系定位（如"购买方"下各子字段）

**通用规范化规则**：

- **日期归一化**：所有日期格式（`2026 年 05 月 15 日`、`2026年5月15日`、`2026-05-15`、`2026/05/15` 等）统一转换为 `YYYY-MM-DD` 存储
- **金额清洗**：去除 ¥、￥、$、,（千位分隔符）及前后空格，负数保留负号
- **校验码清洗**：去除中间空格，连续数字存储
- **全半角统一**：全角字母数字转半角处理

##### E\. 商品明细（特殊处理）

- 商品明细表格转换为 JSON 字符串格式，存入 Excel 单列

- 示例：`[{"货物名称":"电脑主机","规格型号":"标准版","单位":"台","数量":"2","单价":"3500.00","金额":"7000.00","税率":"13%","税额":"910.00"}]`

- **校验规则**（三级校验）：

    1. **商品级**：各商品"金额"（不含税）之和应与主表"不含税金额"一致
    2. **税额级**：各商品"税额"之和应与主表"税额"一致
    3. **合计级**：主表"不含税金额 + 税额 == 价税合计（小写）"成立

    任一校验不通过时在备注列标注具体不一致项（如"商品金额合计 ≠ 不含税金额"、"不含税金额+税额 ≠ 价税合计"）

- **展示示例**：如需展示含税单价信息，可通过 config.yaml 配置扩展字段

- **展开选项**：支持通过 config.yaml 配置将商品明细展开为独立 Sheet（见 3.3.2 说明）

#### 3\.2\.3 成功 / 失败判定规则

- **识别成功**：成功提取「发票号码」且「价税合计（小写）」两个关键字段

- **识别失败**：缺少上述任一关键字段，或 OCR 识别/转换异常

- **输出位置**：控制台实时输出 + 可选日志文件 `processing.log`（通过 config.yaml 配置）

## 4\. 非功能需求

### 4\.1 性能需求

- 单张发票处理时间 ≤ 5 秒（普通清晰度，定义：DPI ≥ 200 或 分辨率 ≥ 1000px）

- 支持单次处理 ≥ 1000 个文件

- **支持并发处理**：通过 `config.yaml` 的 `max_workers` 配置并发线程数（默认 4，设为 0 = 串行）

- 内存占用峰值 ≤ 500MB（串行模式）/ ≤ 2GB（并发模式，max_workers=4）

### 4\.2 可靠性需求

- 异常文件不中断整体批处理流程

- **断点续传**：本版本不实现断点续传（文件级断点续传标记为未来版本功能），但提供处理日志以便手动续传

- **Excel 原子写入**：采用"写临时文件 → os.rename() 覆盖目标"策略，避免写入中途崩溃导致文件损坏

### 4\.3 易用性需求

- 提供命令行一键执行：`python invoice_processor.py ./发票文件夹`

- 提供图形化文件夹选择对话框（可选）

- 详细的错误提示与排错指南

### 4\.4 可扩展性

- 支持新增发票类型（电子普通发票、专用发票、卷式发票等）

- 支持自定义导出字段

- **支持 OFD 格式**（中国电子发票国家标准格式，V1.0 已支持直接解析 OFD 内嵌 XML 结构化数据，准确率 ≈ 100%，优于 OCR）

- **支持直接提取 PDF 内嵌元数据**（部分电子 PDF 发票的 XML 元数据可直接提取，无需 OCR）

## 5\. 异常处理

|异常场景|处理方式|
|---|---|
|文件损坏无法读取|记入【失败处理】Sheet，标注 "文件损坏"（OCR 引擎抛出异常）|
|OCR 识别超时（>30 秒）|终止单文件处理，记入失败|
|PDF 只读加密（可提取文本，禁止编辑/打印）|正常识别处理，日志记录 "PDF 为只读加密，不影响文本提取"|
|PDF 内容加密（需密码打开）|依次尝试 config.yaml 中 `pdf_passwords` 列表内的密码解密；全部失败则记入失败，标注 "加密 PDF（需密码）"|
|OFD 解包失败（非 ZIP 格式）|记录 "OFD 解包失败"，降级至 OCR 或标记失败|
|OFD XML 解析失败|记录 "OFD XML 解析失败"，降级至 OCR（如 fallback_to_ocr 启用）|
|OFD 发票字段缺失|标记缺失字段，其余正常提取，备注列标注 "OFD 字段不完整"|
|Excel 文件被占用无法写入|重试 3 次（可配置），失败则提示用户关闭 Excel|
|磁盘空间不足|提前检测，提示用户清理空间|
|缺少 Tesseract 语言包|友好提示安装命令（参见 2.2 节）|
|重复发票（发票代码+号码相同）|开启去重时仅保留首次成功提取的记录，日志记录 "跳过重复发票（发票代码:xxx 发票号码:xxx）"；未开启则正常处理|

## 6\. 验收标准

### 6\.1 功能验收

1. ✓ 支持递归扫描子文件夹

2. ✓ 正确区分四种发票格式（.ofd/.pdf/.jpg/.png）与其他格式

3. ✓ 成功识别的发票完整提取 21 个字段（含发票类型）

4. ✓ 生成包含 4 个 Sheet 的 Excel 文件

5. ✓ OFD 发票优先走 XML 直接解析，解析失败自动降级至 OCR

6. ✓ 统计数据准确，校验关系成立

7. ✓ 异常文件不中断批处理

### 6\.2 识别准确率验收

- 标准 OFD 电子发票（XML 直接解析）：字段提取准确率 ≈ 100%（结构化数据直取）

- 清晰电子发票 PDF：识别准确率 ≥ 95%

- 清晰扫描件 PDF：识别准确率 ≥ 85%

- 手机拍照发票（光线良好）：识别准确率 ≥ 75%

## 7\. 附录

### 7\.1 配置文件示例（config\.yaml）

```yaml
# 发票批处理工具配置文件
output:
  filename: "发票信息统计.xlsx"
  sheet_success: "成功处理"
  sheet_failed: "失败处理"
  sheet_skipped: "非电子发票"
  sheet_statistics: "处理统计"

ocr:
  engine: paddleocr        # paddleocr（主推）| tesseract | windows_ocr
  lang: ch                 # PaddleOCR 语言模型：ch（中文）| en 等
  use_gpu: false           # 是否启用 GPU 加速
  timeout: 30              # 单文件 OCR 超时（秒）
  enable_vilayout: true    # 是否启用 VI-LayoutXLM 结构化提取

ofd:
  enabled: true              # 是否启用 OFD XML 直接解析
  mapping_path: ""           # OFD XPath 映射文件路径（留空使用内置默认映射）
  extract_preview: false     # 是否提取 OFD 包内预览图片
  fallback_to_ocr: true      # XML 解析失败时是否降级至 OCR

processing:
  max_workers: 4           # 并发线程数（0 = 串行）
  retry_count: 3           # Excel 写入重试次数
  enable_dedup: true       # 是否启用业务去重（以"发票代码+发票号码"为键，同一键值仅保留首次提取记录）
  skip_directories:        # 跳过的目录名称
    - __pycache__
    - node_modules
    - .git
  pdf_passwords:           # 尝试解密 PDF 的密码列表（留空 = 不尝试）
    - ""                   # 空密码（部分电子发票使用）
    - "123456"
    - "password"

extraction:
  strategy: "ofd+xml+ocr"      # ofd+xml+ocr | paddleocr | paddleocr+llm
  llm_fallback: true           # 置信度不足时启用 LLM 兜底
  llm_provider: qwen           # qwen | claude | gemini
  normalize_date: true         # 日期统一为 YYYY-MM-DD
  normalize_amount: true       # 金额去除符号和千位分隔符
  confidence_threshold: 0.85   # 低于此值的字段触发 LLM 兜底或标记人工复核


logging:
  level: INFO              # DEBUG | INFO | WARNING | ERROR
  file: "processing.log"   # 留空则不输出日志文件
```
